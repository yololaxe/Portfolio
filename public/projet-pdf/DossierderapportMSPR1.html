Introduction

Analyse & Cahier des Charges : Préparation et Collecte des Données

Contexte du Projet

Le projet a été conçu pour répondre à un besoin de l’entreprise WildLens, dont l'objectif est de sensibiliser et d'intéresser les individus à la cause animale. Son objectif : détecter des empreintes via une intelligence artificielle capable de reconnaître des caractéristiques visuelles. Cela pourrait non seulement permettre à chacun de mieux connaître la faune environnante,, mais aussi de suivre les déplacements de certaines espèces. Les données collectées pourraient donc présenter un intérêt scientifique notable pour l'examen de ces espèces.

Objectifs

Cette première phase du projet est singulièrement importante en cela que la qualité de sa réalisation impactera directement la totalité de notre projet. Il s’agira, dans un premier temps, de collecter, nettoyer et structurer un jeu d’images labellisés, puis de modéliser une base de données afin de stocker les informations complémentaires sur les espèces et enfin automatiser le processus ETL (Extract, Transform, Load) pour garantir la qualité et l'accessibilité des données.

Sources de Données

Les données nécessaires pour l'application proviennent de deux principales sources :

  • Photos d'empreintes animales : Récupérés grâces à diverses sources iNaturalis, nature-tracking, OpenAnimalTracks, Google Images
  • Données supplémentaires sur les espèces : Fournies dans le fichier infos_especes.csv pour chaque espèce ainsi que d'autres sources récupérées sur le net.

Outils et Méthodologie

        Pour le bon déroulement de ce projet, nous avons utilisé de nombreux outils tels que Selenium, l’API de Google CustomSearch, Pandas, Grafana, PostgresSQL… afin de nous organiser, nous avons appliqué la méthode Kanban avec Trello, nous avons également effectués un diagramme de Gantt. Nous nous sommes donc inspirés des méthodes agiles acquises lors de nos études.

Nous avons mis un point d’honneur à sécuriser la collecte de données tout en la rendant flexible et adaptée pour une automatisation.

Analyse et Nettoyage des Données

Pour cette étape nous avons vérifié la qualité des données afin d'éliminer les erreurs, ensuite nous avons procédé à une augmentation des données pour enfin finir par structurer ces dernières afin de les rendre exploitables..

Modélisation des Bases de Données

Les informations collectées devront être stockées dans une base de données. Une réflexion sera menée sur la structure de stockage la plus adaptée (SQL, NoSQL), afin de garantir l’accessibilité des données pour l'application et les systèmes tiers.

Flux de Données

  • Importation des données : Mise en place des flux pour importer les photos d'empreintes et les informations sur les espèces.
  • Automatisation des requêtes : Préparation des scripts ou requêtes pour la récupération des données dans la base de données en fonction des besoins de l'application.

Livrables

  • Jeu de données d’entraînement, validation et test : Préparation du jeu de données pour l'entraînement du modèle.
  • Modèle de données : Création du modèle de données pour stocker les informations sur les empreintes et les espèces.
  • Script de récupération des données : Création du script permettant d'extraire les informations complémentaires d'une espèce.

Accessibilité Numérique

L'accessibilité des données et des interfaces sera un point clé, avec un focus sur la création de tableaux de bord accessibles à tout public, en prenant en compte les normes d'accessibilité numérique pour les personnes handicapées.

Solution & Stack choisis

Collecte des datas (ETL)

Les outils sélectionnés répondent à leurs besoins spécifiques en matière de collecte, de vérification et d’organisation des données pour WildLens

  • Selenium / WebDriver est un choix préféré pour l’extraction de données des sites Web dynamiques , par exemple, iNaturalist et NatureTracking, qui reposent lourdement sur JavaScript. Contrairement aux librairies statiques comme BeautifulSoup, Selenium permet de simuler les actions des utilisateurs , par exemple, le défilement infini, les clics sur «Charger plus», ce qui est essentiel pour extraire le contenu préchargé par AJAX.
  • Requests / OpenCV : idéaux pour les téléchargements légers , par exemple, des images individuelles, avec un contrôle précis des erreurs , tels que le délai d’attente, le code HTTP
  • Pandas :  Permet la manipulation avancée de métadonnées
  • (fusion de sources variées, élimination des doublons grâce à drop_duplicates()).
  • CSV : Format universel pour l’interopérabilité avec les outils tiers
  •  (Excel, SQL) et la traçabilité des données.
  • Logging : trace les échecs de téléchargements ressources, hiérarchise les erreurs

Ce stack assure la performance et l’auditabilité tout en respectant les contraintes techniques et légales du projet.

Transformation (ETL) grâce à pyspark et panda

L’ETL sera implémenté en utilisant 4 différentes bibliothèques/API Python :

  • PySpark pour traiter de grandes quantités de données
  • Pandas pour lire, manipuler et enregistrer des fichiers CSV ainsi que aligner et structurer correctement les données afin d'assurer leur compatibilité avec Excel.
  • Requests pour effectuer des appels API vers l'intelligence artificielle de Mistral et récupérer dynamiquement des informations sur les espèces animales.
  • Chardet pour détecter l'encodage du fichier CSV avant de le charger.

Visualisation avec Matplotlib & Seaborn

Dans le cadre de l'analyse de notre dataset, nous avons utilisé Matplotlib et Seaborn pour créer des visualisations statistiques avancées. Ces bibliothèques Python offrent une flexibilité et une simplicité d'utilisation, permettant de générer des graphiques informatifs et esthétiques.

  • Matplotlib : C'est la bibliothèque standard en Python pour la visualisation de données. Elle permet de tracer des histogrammes, des boxplots, des barplots, et bien d'autres types de graphiques personnalisés

Seaborn : Basé sur Matplotlib, Seaborn offre une interface de haut niveau pour dessiner des graphiques statistiques attrayants. Il intègre des statistiques avancées et améliore le rendu visuel des graphiques..

Création de la Base de donnée grâce à PostgreSQL (ETL)

Notre projet a besoin d’un système de gestion de stockage des données capable de stocker les résultats finaux (après le modèle d’apprentissage), ainsi que les résultats intermédiaires (Les informations liées au images). Ce système doit se connecter facilement à notre solution ETL et à notre outil de visualisation.

Pour répondre à ces besoins, nous avons testé cinq systèmes de gestion de bases de données souvent utilisés : PostgreSQL, MySQL, MongoDB, SQLite et MariaDB. Notre solution nécessite trois caractéristiques principales de la part d’une base de données :

  1. Gestion des requêtes SQL complexes : Pour de la visualisation des données.
  2. Capacité à contenir une grande quantité de données : Pour stocker les résultats intermédiaires et finaux.
  3. Optimisation pour gérer un grand nombre de requêtes : Pour les transformations, les requêtes de visualisation et l’écriture des données extraites.

Évaluation des options

  • MongoDB : MongoDB est une base de données NoSQL, donc l'exécution de requêtes SQL complexes n'est pas possible, ce qui ne correspond pas à nos besoins de visualisation avancée.
  • SQLite : SQLite n'est pas optimisé pour traiter d'importants volumes de données ou un grand nombre de requêtes simultanées.
  • MariaDB et MySQL : Bien que ces deux peuvent être adaptés, elles offrent des capacités de requêtes moins élevées que celles de PostgreSQL. Leur moteur est également moins optimisé pour gérer un grand nombre de requêtes, ce qui pourrait ralentir notre processus de transformation et de visualisation des données.

Dockerisation avec Docker

Pour faciliter le déploiement de notre solution, nous avons choisi d'utiliser Docker. La dockerisation permet de créer des conteneurs pour chaque composant de notre stack (Python, PySpark, PostgreSQL, Grafana), assurant ainsi une installation et une mise à jour simplifiées. Cela garantit également la portabilité de notre solution sur différents environnements, que ce soit en développement, en test ou en production (Surtout pour la partie 2 du projet).

ETL : Récupération des données (Extract)

Introduction

Le processus ETL (Extraction, Transformation, Chargement) est essentiel pour préparer les données relatives aux animaux, à leurs empreintes et à leurs caractéristiques, pour ensuite faire l'entraînement du modèle de reconnaissance d'empreintes animales.

  1. Les images d'empreintes

L’idée, afin de tester au mieux la qualité de notre code, a été de récupérer nos données depuis diverses sources :

  • Une source très fiable, OpenAnimalTracks : Trouvée grâce au site arxiv.org et récéptionnée auprès de ses auteurs suite à une demande par mail.
  • Des sources fiables, contenant possiblement des erreurs : Il s’agit ici des sites iNaturalist et Nature Tracking dont les données sont fournies et revues par des utilisateurs, il peuvent donc contenir quelques rares erreurs, mais cela reste intéressant.
  • Des sources moins fiables : Il s’agit de Google Images, l’intérêt était ici triple :
  • Accéder à une plus grande diversité de données
  • Cibler la récupération de données en fonction d’un animal
  • Ajouter du bruit à nettoyer afin de tester au mieux notre nettoyage de données visuel.

→ Cette dernière option impose une vérification préalable des domaines de sites afin de ne récupérer que des images libres de droit depuis des sites dont le fichier robots.txt ne l’interdit pas explicitement. Nous avons implémenté une version rudimentaire de ce tri en fonction des sites mais cette dernière n’est probablement pas suffisante, ainsi les données utilisées dans ce cadre, le sont grâce à l'exception pédagogique en France (Article L122-5 du Code de la propriété intellectuelle) et ne seront pas utilisés pour l’entraînement du modèle, mais seulement lors du test et de l’évaluation de notre code.

Concernant l’organisation du projet, nous avons un fichier pour chaque source, i_naturalist.py pour iNaturalist, nature_tracking.py pour Nature Tracking et enfin scrap_google.py pour Google Images.

La collecte de ces données est sécurisée grâce à l’ajout de délais pour respecter les règles des sites, des validations HTTPS, des filtres de domaines…

             Exemple de filtrage dans le fichier scrap_google.py

La fonction check_robots_permission par exemple permet de vérifier les permissions spécifiées dans robot.txt pour les respecter. De plus, une validation des fichiers est effectuée au travers du filtrage des formats jpeg et png, d’une détection des fichiers corrompus via l’ent-tête des images et enfin une taille maximale de 10Mo.

La gestion des erreurs est effectuée au travers d’une relance automatique des requêtes en cas de tentatives échouées (3 tentatives) et un délai de 2.5 secondes entre les requêtes pour éviter le bannissement. Enfin les liens non sécurisés (HTTP) sont filtrés et un proxy configurable est mis en place pour anonymiser les requêtes.

Le fichier scrap_google.py, afin de récupérer des données plus diversifiées permet d’introduire des méthodes qui seront par la suite développées plus en détail par l’ETL.

Nous avons par exemple la vérification des doublons, afin d'éviter de télécharger deux fois une même image avec la fonction load_existing_urls().

De plus, une détection visuelle est effectuée grâce à OpenCv pour détecter les contours d’empreintes.

Dans cette fonction, on décode l’image avec OpenCV, on la convertit en niveaux de gris (cv2.cvtColor()) et on y ajoute un flou gaussien (cv2.GaussianBlur()) afin d’analyser au mieux l’image. La détection des contours est effectuée avec cv2.Canny() puis un filtre est appliqué pour ne garder que les contours supérieurs à 1000 pixels afin que seul les images les plus susceptibles de contenir des empreintes d’animaux soient gardés.

Les fichiers relatifs à iNaturalist et Nature Tracking sont eux bien différents. Leurs scripts respectifs se concentrent essentiellement sur l’automatisation d’action sur les pages afin de charger toutes les données avant de d’extraire l’url des images ciblées ainsi que le nom de l’animal associé.

La plupart de ces scripts sont relativement modulaires et peuvent aisément être appelés en local via un fichier externe ou en consommant une API, enfin, le fichier sort_data.py nous a permis de trier et normaliser les données afin de pouvoir les transmettre directement à notre base de données.

Au total, nous avons récupéré 940 images labellisées prêtes à être utilisées pour l'entraînement de notre modèle grâce à nos scrappers. Cela additionné aux 2 335 images récupérées avec le dataset Open Animal Tracks, fait un total de 3 275 images à envoyer à l’ETL.

  1. Les Métadonnées (Information sur les mammifères)

Récupération :

L’objectif ici est de garnir du maximum possible le fichier CSV des noms des animaux mammifères pour accroître notre base de données d’animaux identifiables. Pour cela plusieurs sources fiables ont été utilisé :

Infos_especes : Liste d’espèces d’animaux fournir par l’école et qui sert de guide aux informations des espèces à recueillir

Animal information dataset (kaggle) : Cet ensemble de données englobe un large éventail d'attributs relatifs à diverses espèces animales du monde entier. L'ensemble de données comprend principalement des champs tels qu'Animal, Taille (cm), Poids (kg), Couleur, Durée de vie (années), Régime alimentaire, Habitat, Prédateurs, Vitesse moyenne (km/h), Pays trouvés, Statut de conservation, Famille, Période de gestation (jours), Vitesse de pointe (km/h), Structure sociale et Progéniture par naissance.

Diconimoz : Le site possède une liste des mammifères sauvages classés de A à Z, couvrant diverses espèces terrestres.

Instinct Animal : Le site propose une liste des mammifères classés par ordre alphabétique de A à Z. Chaque animal comporte une fiche descriptive complète et détaillée avec des photos et des illustrations.

L’ensemble des informations de ces sources ont été récupérées puis stockées dans un même fichier csv.

A présent nous disposons d’une grande liste de noms d’espèces d’animaux mammifères mais nous ne disposons pas de la totalité des informations à savoir la description, le nom latin, la famille, les tailles minimales et maximales, la région, l’habitat et le fun fact de ces animaux.

  1. Automatisation du remplissage des données animalières

Pendant le développement, plusieurs problèmes techniques se sont présentés, et tous ont été résolus au fur et a mesuré résolu.

  1. Problèmes rencontrés et solutions mises en place

Encodage des caractères spéciaux mal géré

Problèmes identifiés : Les accents et caractères spéciaux (ex : é, à, ç) étaient mal affichés sous Excel, générant des symboles incompréhensibles.

Des erreurs de lecture survenaient (« UnicodeDecodeError »), ce qui bloquait l'ouverture du fichier.

Solutions mises en place :

  • Utilisation de l'outil Chardet pour pouvoir identifier l'encodage du fichier source.
  • La lecture du fichier avec l’encodage adapté (ISO-8859-1 ou UTF-8).

·         Enregistrement final en UTF-8-SIG, qui assure une compatibilité optimale avec Excel.

 

Mauvaise structuration des colonnes dans Excel

Problèmes identifiés :

Toutes les données étaient fusionnées dans une seule colonne.

Le séparateur de colonnes (« , » ou « ; ») était mal interprété.

Solutions mises en place :

  • Détection automatique du séparateur correct (; ou ,).
  • Sauvegarde du fichier CSV en forçant le bon délimiteur (;), pour garantir un affichage fluide dans Excel.

Mise en place de Docker Compose:

Nous avons mis en place la base de données PostgreSQL en utilisant Docker grâce à un fichier docker-compose.yaml .Ce document renferme tous les réglages essentiels pour paramétrer le service PostgreSQL, établir un volume de données durable et modifier les configurations réseau.Cette configuration garantir simultanément la protection des données et leur accessibilité.

 

Accès à la base de données en utilisant DBeaver

Pour simplifier la gestion de notre base de données, nous avons opté pour DBeaver. L'établissement de la connexion a la base de données PostgreSQL s'est effectué en suivant des étapes simples , en fournissant les informations de connexion telles que l’hôte, le port, le nom de la base de données l'utilisateur et le mot de passe.

 

 

Création des tables

La création des différentes tables a été effectuée grâce à une suite de requêtes SQL exécutées directement via DBeaver.


Représentation du modèle relationnel :

 

Intégration des données

Pour alimenter la base de données avec des données sur les animaux, nous avons utilisé des instructions "INSERT INTO".
Ces requêtes facilitent l'insertion d'informations précises, telles que les noms des animaux, leurs caractéristiques, leurs habitats et même les photos associées à leurs empreintes.


Accès aux données

Nous avons effectué des requêtes SELECT* sur les tables dim_animal et dim_image. Ce qui nous a permis de valider que les informations étaient bien enregistrées et qu'elles apparaissent comme prévu dans DBeaver.

 

 

 

Conclusion

Grâce à l'utilisation de Docker pour le déploiement de la base de données sur DBeaver, nous avons pu établir une base de données structurée. L'intégration des données dans la table dim_animal a permis la mise en place de la base de données.
